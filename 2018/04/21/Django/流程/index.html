Django wsgi启动流程

Posted on 2018-04-21 | In django

在Web应用启动后，django.setup()加载settings,配置logging,加载app！
之后会生成一个 WSGIHandler 实例,也就是WSGI application接口实现的可调用对象
在实例化WSGIHandler的__init__方法中,完成加载中间件
setting中的WSGI_APPLICATION = ‘project.wsgi.application’ 与此配置相关
之后每次由WSGI server转发来的请求都会使用这个实例进行响应(当然也可能同时存在多个application实例:与WSGI服务器的配置有关)

对于每次请求,WSGI server调用WSGIHandler的__call__方法

  • 1.生成WSGIRequest实例,发送request_started信号
  • 2.进入中间件的链式调用,loops through the request methods, running them in order
  • 3.resolves the requested URL 处理url
  • 4.loops through each of the view processing methods
  • 5.calls the view function (usually rendering a template),也就是url模块中的as_view
  • 6.processes any exception methods
  • 7.loops through each of the response methods, (from the inside out, reverse order from request middlewares) 注意：这里的顺序是逆序
  • 8.calls the callback function to the web server

WSGIHandler

wsgi.py定义了获取application的方法, 代码从django.core.wsig引入了

跳转到get_wsgi_aoolication看一看,首先调用setup()方法,初始化Django,加载app等
get_wsgi_application() 其实返回一个WSGIHandler实例

WSGIHandler类继承自BaseHandler,实现了__call__方法,作为WSGI server的回调函数, 这就是request请求进到Django的入口.
那么WSGIHandler是怎么实现的,看一下源码,WSGIHandler的init方法调用了父类方法,并执行了load_middleware加载中间件

Middleware类

注意：Django版本1.11
中间件的加载在实例化WSGIHandler的过程中实现(__init__), self.load_middleware()

Django 1.10版本后的middleware接口有变化,具体看文档
实现类接口的自定义MiddleWare类,继承自MiddlewareMixin类,实例化中间件是需要指定get_response的可调用对象,该可调用对象又在类的__call__中调用，t通过将上一级的中间件实例作为init的实参,这样就可以实现一系列中间件的链式调用,loading_middleware执行结束，加链式调用中间件的入口绑定到self._middle_chain属性上，该属性将会在appliacation的get_response方法中调用

注意,中间件列表的最后一项的get_response 方法指向WSGIHandler类的_get_response方法, 这个方法是django url/view/db/template等模块处理请求的入口

当一个中间件的process_request方法返回HttpResponse对象（或者非None）,程序将会跳过之后所有settings.MIDDLEWARE中该中间件之后的其他中间件定义的所有方法,直接进入到返回HttpResponse对象的中间件及settings.MIDDLEWARE中该中间件之前的所有中间件的process_reponse方法(逆序)

WSGI Application

回到WSGIHandler, 重新看一下该类的__call__方法:

WSGIHandler 处理器准备工作已经完成,当被WSGI_server调用时,进入call方法,随后它给调度程序发送一个信号 request_started,然后根据入 environ 构造 WSGIRequest 对象,它的父类是HttpRequest.

接下里调用get_response(request), 看一下原代码,这里面会调用中间件链式调用过的入口函数,得到响应,对响应做一些处理,调用start_response发送响应头,之后返回响应体,至此，对一个请求的响应结束

那么get_response方法中又做了哪些事情

首先设置urlconf, 其后，调用self._middleware_chain得到response

此时先按照中间件正序调用所有的process_request,如果该方法返回None,之后调用_get_response方法,

这里可以清晰的看到各个中间件函数的执行顺序

首先解析url,找到处理该url的视图view，之后应用中间件process_view函数,然后调用view处理请求,对view 的异常进行捕获,使用异常中间件进行处理，之后返回response,调用中间件的process_response函数处理,最后回到WSGIHandler的call函数完成整个请求处理过程.
